# 0002: Design review — pre-implementation fixes and threshold registration

Date: 2026-07-19

## Context

A full review of `DESIGN.md` and `IMPLEMENTATION_PLAN.md` (against `MARKET_RESEARCH.md`, the agent definitions, and both project skills) before any code exists. The architecture, phase gates, and risk controls validated; four blocking document defects and several measurement-validity gaps were found and fixed in the same change as this record.

## Decisions

### 1. Validation thresholds pre-registered (owner-set 2026-07-19, before any data collection)

- **H1 (existence):** ≥ **20** net-of-fee opportunities/week across all approved pairs. Also pinned as N in MR §11's decision gate.
- **H5 (economics):** annualized hurdle rate **10%** (after the 20% slippage haircut).
- H2–H4, H6 already carried numeric thresholds; unchanged.
- H6 now participates in the gate: VALIDATED requires H1–H6; H1–H5 pass with H6 failing is WEAK SIGNAL (extend, don't build).

Per CLAUDE.md invariant 8, no future change may touch these values in the same change that computes results.

### 2. Golden fee cases derive from reconciled live schedules, not the literature anecdote

MR §3.4's "$3.15 fees on a 2¢ × 100-contract spread" example is from the Kalshi↔Polymarket **Global** pair and is not derivable from the documented Kalshi + PM-US schedules (which peak at ~$3.00 combined at 50¢). It was slated to become a *permanent-invariant* golden test — an unreproducible frozen expectation that would have deadlocked M1.

**Open item (blocks M1 coding):** api-researcher must produce `docs/decisions/0003-fee-schedule-reconciliation.md` resolving:
- Polymarket US updated its fee schedule effective **July 1, 2026** ([docs.polymarket.us/fees](https://docs.polymarket.us/fees), [help center](https://help.polymarket.com/en/articles/13364478-trading-fees)) — MR §3.3's April 3 numbers (0.05 coefficient, $1.25/100 cap) vs the "flat 0.30% taker / 0.20% maker rebate" description need reconciling against the live fee endpoint.
- Kalshi maker fee: MR §3.1 says "flat 0.25% per contract on some markets"; current sources describe "25% of the taker fee" (≈ `0.0175 × P × (1−P)`) ([Kalshi help](https://help.kalshi.com/en/articles/13823805-fees), [fee schedule PDF 7.7.26](https://kalshi.com/docs/kalshi-fee-schedule.pdf)). These differ; the per-fill round-**up** behavior must be confirmed alongside.
- Rounding directions: Kalshi rounds up per fill; PM-US documents nearest-cent rounding. Golden cases pin both.

Kalshi rate-limit facts in MR §4.1 (token bucket 2026-04-23, Basic 200/100 tokens/s at 10/request, no Retry-After) were independently confirmed against [docs.kalshi.com](https://docs.kalshi.com/getting_started/rate_limits) — no change needed.

### 3. Pair-approval scope unified ("used" defined)

Three documents carried three different scopes; the strictest (security-review: no pair "fetched, subscribed, recorded, or scored" without approval) made M2's probe and M3's matcher self-blocking. Unified definition, now identical in CLAUDE.md invariant 4, DESIGN §4.2's context, and the security-review skill: **discovery/scoring is always allowed; recording, paper-trading, and trading require an approved registry entry; M2 may stream one explicitly designated probe pair pre-registry.**

### 4. Execution-path scaffolding rules aligned

- `VenueClient` (DESIGN §4.1) no longer declares venue-write methods; a `TradingVenueClient` extension protocol is specified only when first permitted.
- Standing rule 3 rewritten: no venue writes during phase 0; sole later exception is the Kalshi **sandbox** order-lifecycle harness as the final phase-1 pre-live checklist item; PM-US minimum-size live-order validation is the first phase-2 step.
- DESIGN §9.4's checklist re-labeled to match (it gates G1 → G2, built end of phase 1).

### 5. Measurement-validity additions (DESIGN)

- Detector staleness guard (`max_book_age_ms`, default 1500) — a stale book suppresses detection; spread events record both books' ages (phantom-spread audit).
- Paper trader adds venue-imposed `seconds_delay` to the latency model; results split pre-game vs in-game.
- Fee params resolved per market/series (`fee_class` on `MarketMeta`) — Kalshi rates vary by series.
- PM-US WS capacity carries the market-vs-token instrument uncertainty (3 vs 6 connections for 30 pairs); M2 day-1 probe resolves it.
- H1 reported both raw and normalized per covered game-hour (downtime-exclusion bias); August-window seasonality caveat recorded in §9.2.
- Recorder flush cadence made explicit (`flush_interval_s` ≤ 30, write-temp-then-rename); `spreads` schema now stores the book timestamps/ages used.

### 6. Smaller plan fixes

- M0 gains a VPS-provisioning owner task (M4's soak had no provisioned target).
- `.claude/settings.json` gains the allow-list M0 specified (`uv run pytest/ruff/mypy`).
- Pivot-path wording corrected: recorder/detector/fee infra is category-agnostic, the matcher is not (league/team/start-time keys); a pivot needs a rules-text matcher and fresh resolution-risk review.
